Audit sicurezza e tenant isolation GPT

Codebase: TouraCore Data audit: 2026-04-26 Scope: Next.js app, packages core, Supabase migrations, API pubbliche, cron, tenant/RLS

Executive Summary

La postura generale e buona per una piattaforma multi-tenant: RLS e presente in modo esteso, le query applicative principali filtrano per tenant/entity, le route critiche usano guard espliciti, i webhook Stripe sono firmati e deduplicati, e le view KPI multi-tenant sono state corrette a security_invoker=true.

Il rischio residuo non e nel modello tenant di base, ma nelle eccezioni operative: route cron pubbliche che in alcuni casi fidano un header spoofabile, API key con scope salvati ma non applicati negli endpoint v1, key pubbliche accettate in query string, e rate limit in-memory non adatto a produzione serverless/multi-instance.

Findings Prioritari

High SEC-001 - Alcune cron route accettano x-vercel-cron come bypass del segreto

Alcune route cron sono escluse dall'autenticazione middleware e accettano la presenza dell'header x-vercel-cron come autorizzazione completa. Se un request esterno puo inviare quell'header fino all'app, il CRON_SECRET viene bypassato e operazioni con service role possono essere avviate.

Evidenze: packages/core/auth/src/middleware.ts:66 marca /api/cron/ pubbliche; apps/web/src/app/api/cron/accounting-sync/route.ts:10 e apps/web/src/app/api/cron/pricing-suggestions/route.ts:11 fanno return true solo per header presente.

Impatto tenant isolation: alto impatto operativo cross-tenant, perche molte cron iterano tutti i tenant/entity con service role.

Remediation: richiedere sempre Authorization: Bearer CRON_SECRET o una firma HMAC; usare x-vercel-cron solo come segnale diagnostico, non come auth. Centralizzare un helper assertCronAuthorized(req) e sostituire tutte le implementazioni locali.

High SEC-002 - Scope delle API key v1 non applicati dagli endpoint

authenticateApiKey legge e ritorna scopes, ma gli endpoint v1 controllano solo la validita della key e il tenant. Una key attiva del tenant puo leggere reservations, room types o proprieta anche se non dovrebbe avere quello scope.

Evidenze: packages/core/api/src/auth.ts:31 seleziona scopes e packages/core/api/src/auth.ts:45-49 li ritorna; apps/web/src/app/api/v1/reservations/route.ts:6-7 autentica senza verificare scope prima della query su reservations.

Impatto tenant isolation: tenant isolation e mantenuta via ctx.tenantId, ma l'isolamento per capability/partner/API consumer e debole.

Remediation: introdurre authenticateApiKey(req, requiredScope) o assertScope(ctx, 'reservations:read'); aggiungere test negativi per key senza scope su ogni endpoint.

Medium SEC-003 - Public booking key accettata in query string

extractKey accetta la chiave anche da ?key=.... Le query string tendono a finire in log, analytics, history browser e referrer. Per chiavi che autorizzano booking/checkout, questo aumenta il rischio di leakage.

Evidenza: apps/web/src/app/api/public/booking/_shared.ts:99-104.

Remediation: mantenere solo X-Booking-Key o Authorization; se serve retrocompatibilita, deprecare la query key con logging e rotazione automatica delle key usate in URL.

Medium SEC-004 - Rate limit in-memory non sufficiente in produzione distribuita

Il rate limiter usa una Map in memoria del singolo processo. In serverless o multi-instance il limite si resetta per istanza e non protegge bene endpoint costosi o brute force distribuiti.

Evidenze: packages/core/security/src/rate-limiter.ts:8, packages/core/security/src/rate-limiter.ts:25-40.

Remediation: usare store centralizzato (Upstash Redis, Vercel KV/Firewall, Supabase table con TTL) e includere protezioni specifiche per auth, AI, public booking e partner API.

Medium SEC-005 - Endpoint AI autenticato ma non tenant-scoped ne quota-scoped

suggest-reply richiede solo un utente autenticato e poi consuma Anthropic con input arbitrario. Non verifica che propertyName appartenga a un tenant/entity accessibile e non applica quota per tenant/utente.

Evidenze: apps/web/src/app/api/v1/ai/suggest-reply/route.ts:71-82.

Impatto tenant isolation: non ho visto data leak diretto perche il contenuto e user-supplied, ma l'endpoint puo diventare abuso costo/brand e aggirare limiti per tenant.

Remediation: richiedere entityId o reviewId, verificare via RLS/guard, contabilizzare usage per tenant e applicare quota/rate limit persistente.

Medium SEC-006 - CSP consente unsafe-inline in produzione

La CSP include script-src 'unsafe-inline' anche fuori dev. Questo riduce l'efficacia della CSP come barriera anti-XSS, specialmente con superfici che renderizzano HTML/sanitized tags.

Evidenza: packages/core/security/src/headers.ts:14.

Remediation: migrare a nonce/hash per script inline dove possibile, mantenendo fallback controllato solo per route che lo richiedono davvero.

Low SEC-007 - Confronto hash public key non constant-time

Il commento promette confronto timing-safe, ma il codice usa hash !== row.key_hash. Il rischio pratico e basso per hash SHA-256 hex fissi e accesso remoto rumoroso, ma e una discrepanza facile da correggere.

Evidenze: apps/web/src/app/api/public/booking/_shared.ts:16 e apps/web/src/app/api/public/booking/_shared.ts:41-42.

Remediation: usare crypto.timingSafeEqual su buffer di uguale lunghezza.

Cosa Va Bene

  • Stripe webhook usa firma ufficiale e dedup evento prima della mutazione.
  • Env validation richiede secret forti in produzione.
  • CSRF token ha confronto constant-time lato helper.
  • Route dashboard principali passano da middleware auth e bootstrap Supabase con RLS.
  • Le server action restaurant campionate verificano ownership prima di usare service role.
  • Il repository contiene gia test e2e per scope cross-entity e audit flows.

Piano 30/60/90

0-7 giorni

  • Rimuovere bypass x-vercel-cron o proteggerlo con secret.
  • Applicare scope API key su tutti gli endpoint /api/v1/*.
  • Disabilitare key in query string per nuove public booking key.

30 giorni

  • Rate limit persistente per auth, AI, booking e partner API.
  • Quota AI per tenant/utente e audit usage.
  • Test automatici negativi cross-tenant e no-scope API.

60-90 giorni

  • Riduzione service role con RPC dedicate o client RLS dove possibile.
  • CSP con nonce/hash e inventario delle route che richiedono inline script.
  • Controllo periodico migrazioni per nuove tabelle senza RLS o view senza invoker.

Conclusione

Verdetto: buona base multi-tenant, non ancora hardening completo per produzione ad alto rischio. La tenant isolation dati e progettata correttamente nel database e spesso ribadita nell'app, ma le superfici operative pubbliche devono essere strette prima di considerare il sistema robusto contro abuso esterno e privilege expansion per API consumer.
