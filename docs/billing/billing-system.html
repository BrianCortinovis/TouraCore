01Riassunto in 60 secondi

Cliente prenota su villasole.touracore.it. Inserisce carta. Niente viene addebitato subito (rate plan free_cancellation di default).

7 giorni prima del check-in, un cron automatico crea un PaymentIntent direttamente sull'account Stripe del tenant con application_fee_amount = commissione TouraCore.

Al check-in, l'hotel preme "Cattura": Stripe splitta i soldi alla fonte. €900 vanno all'hotel, €100 a TouraCore. Mai un euro transita sul conto TouraCore.

Stripe bonifica daily al tenant via SEPA. TouraCore riceve i propri €100 sul proprio IBAN.

02Flusso fisico dei soldi

La regola architetturale: TouraCore mai banca. Stripe Connect Direct Charge splitta alla fonte.

Garanzia regola architetturale Nessun checkout pubblico viene attivato senza tenant Stripe Connect charges_enabled = true. Block hard a livello server action di pubblicazione entity (STRIPE_CONNECT_REQUIRED).

04Le 4 tariffe (rate plans)

Il tenant configura quali tariffe offrire in /[tenantSlug]/{stays|dine|rides|activities}/[entitySlug]/rate-plans. Default auto-creato alla prima visita = free_cancellation.

TipoQuando addebitaCancellazioneTipico per
free_cancellation Carta salvata al booking. Charge 7gg prima del check-in. Gratuita fino a 7gg prima. Default. Hotel/B&B/esperienze flessibili.
deposit_30 30% subito + saldo 30gg prima. Gratuita fino a 30gg prima, poi perdi il 30%. Strutture con cash flow esigente.
partially_refundable_50 50% subito + saldo 14gg prima. Rimborso 50% se cancelli. Bilanciato impegno cliente / flessibilità.
non_refundable 100% subito (sconto 10%). Nessun rimborso. Offerte Early Bird / non-refundable rate.
Stessi 4 tipi vale per tutti i 4 verticali Hospitality, restaurant, bike, experience usano la stessa tabella payment_rate_plans con flag vertical. UI condivisa RatePlansEditor riusata.

05Timeline prenotazione → pagamento

Esempio rate plan free_cancellation, prenotazione 6 mesi prima del check-in.

  1. Mese 0 — Cliente prenota. Inserisce carta sul widget pubblico. Stripe SetupIntent salva token off_session. Niente addebito. Niente blocco plafond. Stato: payment_state = card_saved.
  2. Mese 1-5 — Attesa. Niente. Cliente vede zero movimenti. Tenant vede prenotazione confermata. Stripe Card Updater rinnova automatico carta se scade.
  3. T-7gg — Cron auto-charge. auto-charge-bookings alle 03:00 UTC trova la prenotazione, crea PaymentIntent off_session con capture_method = manual. Stripe pre-autorizza €1000. Stato: payment_state = authorized.
  4. Giorno check-in — Capture. Hotel preme "Cattura" (o cron lo fa al check-out). Stripe esegue split: €900 al tenant + €100 fee TouraCore. Stato: payment_state = captured.
  5. Stripe payout daily. €900 dal balance tenant → IBAN tenant via SEPA. €100 dal balance TouraCore → IBAN TouraCore.
Limite Stripe pre-auth = 7 giorni Per questo per booking >7gg in anticipo serve la fase 1 (carta salvata) + fase 3 (charge a T-7gg). Stripe non permette pre-auth più lunghe; carte salvate sì, indefinitamente.

10Cosa succede se la carta fallisce

Sistema di recupero a 3 livelli.

  1. T-7gg fallimento 1 — cron auto-charge-bookings riceve insufficient_funds. Marca payment_state = failed. Schedula retry +24h. Webhook payment_intent.payment_failed registra attempt.
  2. T-6, T-5, T-4 — retry 2/3/4. Stesso ciclo. Se cliente nel frattempo ricarica carta → success. Se 4 fail consecutivi → niente più retry.
  3. Magic link "aggiorna carta". Email cliente con URL /r/[token]/update-card (HMAC firmato 7gg). Cliente clicca → redirect Stripe Customer Portal nativo → aggiorna carta → token rimpiazzato sul Customer → prossimo cron usa la nuova.
  4. T-3gg — auto-cancel. Se ancora failed e nessun retry pending, cron auto-cancel-failed-payments marca payment_state = auto_cancelled + status = cancelled. Camera/slot liberato. Niente refund (mai addebitato).
Il vantaggio dell'addebito differito Niente è mai stato spostato → niente da rimborsare. Stripe fee zero su prenotazioni che non si concretizzano. Tenant ha tempo di rivendere lo slot.

11Tabelle DB chiave

tenants.stripe_connect_* (migration 00149)
Colonne aggiunte a tenants: stripe_connect_account_id, stripe_connect_charges_enabled, stripe_connect_payouts_enabled, stripe_connect_details_submitted, stripe_connect_country, stripe_connect_requirements (JSONB). Mirror dal webhook account.updated.

supabase/migrations/00149_tenants_stripe_connect.sql
payment_rate_plans (migration 00150)
Tariffe per entity. Colonne: vertical, type, refund_window_hours, deposit_pct, discount_pct, charge_balance_days_before, is_default, active. UNIQUE su (entity_id) WHERE is_default. RLS: tenant write+read, anon read se active=true, service-role full.

Nome payment_rate_plans per non collidere con legacy rate_plans hospitality.

supabase/migrations/00150_rate_plans_and_payments.sql
reservation_payment_methods (migration 00150)
Token Stripe carta salvata per reservation (polimorfica via vertical+reservation_id). Colonne: stripe_customer_id, stripe_payment_method_id, stripe_setup_intent_id, card_brand/last4/exp/funding/country, is_primary.
reservation_payment_attempts (migration 00150)
Audit retry. Per ogni tentativo charge: attempt_number, stripe_payment_intent_id, amount_cents, status, failure_code, failure_message, retry_at. Index parziale per cron retry lookup.
Colonne payment state su 4 reservation tables
Aggiunte su reservations (hospitality), restaurant_reservations, bike_rental_reservations, experience_reservations:
  • payment_rate_plan_id (FK)
  • charge_scheduled_at (TIMESTAMPTZ, indice parziale per cron)
  • payment_state ENUM: pending / card_saved / authorized / captured / failed / auto_cancelled / refunded
  • application_fee_amount_cents (snapshot fee al momento del checkout)
